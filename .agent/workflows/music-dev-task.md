# 亲亲音乐 开发任务工作流

当用户通过 `/music-dev` 触发本工作流时，进入标准开发任务执行模式。本工作流适用于亲亲心情笔记项目（`music_community_mvp`）的所有功能开发与 Bug 修复任务。

---

## 触发方式

**指令格式**：`/music-dev [任务编号] [可选描述]`

| 指令示例 | 含义 |
|---------|---------| 
| `/music-dev T19 资源压缩` | 执行 T19 资源管道化处理任务 |
| `/music-dev T20-3 评论分页` | 执行 T20 的第 3 个子任务 |
| `/music-dev T27-1 拆分 profile_view` | 执行代码质量提升的第 1 个子任务 |

**项目路由表**（用于自动定位文件路径）：

| 项目 | 工程根目录 | 开发任务列表 | 测试清单 | 变更日志 | 代码变更记录 | 进度看板 | 测试报告目录 |
|------|-----------|------------|---------|---------|-------------|---------|------------|
| 亲亲音乐 | `music_community_mvp/` | `docs/dev/开发任务清单-v2.0.md` | `docs/test/测试清单-v2.0.md` | `docs/Changelog.md` | `docs/CodeChangeLog.md` | `docs/TaskBoard.md` | `docs/test_reports/` |

## 【红线规则 — 最高优先级】

### 🔴 物理写盘红线 (Anti-Hallucination)

整个流程中要求「生成」「创建」「保存」的所有文档和代码，**绝对禁止仅在对话框中输出文字预览**。你**必须全程主动调用**真实的文件读写工具（`write_to_file`、`replace_file_content`、`multi_replace_file_content`）将结果**真实写入物理磁盘**。如果你仅在回答中展示文本内容而未触发写盘工具，视为严重流程故障。

### 🔴 强制预读红线 (Context Loading)

在动手写任何代码之前，**必须先阅读**以下文档来恢复完整上下文：
1. **AGENTS.md** — 了解红线规则和代码约束
2. **README.md** — 了解项目全貌和技术架构
3. 对应版本的**开发任务列表**（确认任务编号与范围）
4. 对应版本的**测试清单**（了解将如何被验证）
5. 与本次任务**直接相关的已有源代码文件**（避免重复造轮子或破坏已有逻辑）
6. 对应版本的**PRD 文档**中与本次任务对应的章节（确认业务规则）
7. 如涉及数据库变更：**sql/ 目录下相关的 SQL 文件**
8. **精准预读技术图纸**：如果该任务带有 `[Tech Spec 必须]` 标记，**仅加载** `docs/tech_specs/v{当前版本}/` 目录下与本任务编号匹配的 **那一份** 技术方案文档（如 `TS-T28...md`）。**绝对禁止**一次性读取 `tech_specs` 下的所有文档。

> **📁 归档感知规则**：优先读取 `docs/changes/` 目录下最近的变更档案来获取近期上下文。
> 坚决**不要**主动全量加载 `docs/archive/` 下的历史文件。
> 归档文件仅在排查历史 BUG、跨版本溯源时按需读取。


### 🔴 变更级归档红线 (Change-Level Archiving)

每完成一组有完整意义的代码变更，**绝对禁止**追加到单体 `Changelog.md` 中，必须在 `docs/changes/` 目录下生成一个独立的 Markdown 文件（命名格式：`YYYYMMDD-[任务编号]-[简述].md`）。
文件格式需严格遵照 `.agent/skills/changelog_writing/SKILL.md` 规范。

### 🔴 代码约束红线

- **单文件不超过 300 行**，超过必须拆分为子组件
- **【针对巨型文件的拆分 (如T27)】必须先在设计阶段列出子组件拆分结构，确认后逐个文件单次写入，绝对禁止试图一次性替换整个 400 行以上的超大文件。**
- **Supabase 调用必须封装在 `data/services/` 层**，Controller 和 View 禁止直接调用
- **每个 Supabase 调用必须有 try-catch**，并使用 LogService 记录错误
- **涉及 RLS 策略变更时，必须先在 `sql/` 目录创建对应的 .sql 文件**
- **禁止在循环中发起 Supabase 请求**
- **禁止使用 `.count()` 客户端调用做统计**，改用 RPC

## 执行步骤

### 阶段 1：任务定位与上下文加载

1. **解析用户指令**
   - 解析 `/music-dev [任务编号]` 格式。
   - 根据项目路由表定位到正确的文档路径。
   - 如未指定明确的任务编号，**立即询问用户**：「请问要执行哪个任务？（参考 docs/dev/开发任务清单-v2.0.md）」

2. **强制预读文档**（不可跳过）
   - 阅读 `AGENTS.md`，刷新红线规则。
   - 阅读对应版本的开发任务列表，定位当前任务的具体要求。
   - 阅读 PRD 中相关章节，确认业务规则。
   - 阅读已有的相关源码文件，理解当前代码结构。
   - 如涉及 Supabase 表结构变更，阅读 `sql/` 目录下的相关 SQL 文件。

3. **在 TaskBoard 登记**
   - 打开 `docs/TaskBoard.md`。
   - 在对应位置将当前任务标记为 `🔄 进行中`。

### 阶段 2：技术方案前置校验 (Tech Spec Check)

4. **拦截无图纸施工**
   - 检查该任务在开发清单中是否带有 `[Tech Spec 必须]` 标记，或是否属于高复杂度的架构重构、性能优化。
   - 如果属于高复杂度任务且尚未生成对应的 `docs/tech_specs/` 文档，**必须立即中断开发流程**，并向用户报警：“该任务属于高风险重构，必须先进行技术方案评审。请使用 `/music-tech-spec [任务名]` 指令呼叫架构师进行方案选型。”
   - 如果已存在对应的 Tech Spec 文档，则必须严格按照该文档的【实施拆解】进行编码，绝对禁止私自篡改架构设计。
   - 如果是简单的增改逻辑（无系统风险），则直接进入编码阶段。

5. **编码实现**
   - 在**真实的工程目录**中进行代码编写。
   - 所有新增的类、函数必须包含文档注释（说明用途、参数及返回值）。
   - Supabase 数据操作必须封装在 `data/services/` 层。
   - 每完成一个独立模块/文件，立即自查是否超过 300 行限制。
   - 如涉及 Supabase 表结构变更，先在 `sql/` 目录创建 SQL 迁移文件。

6. **执行变更级归档** ← 📌 绝对不可遗忘
   - 编码完成后，查阅 `changelog_writing` 规范。
   - 在 `docs/changes/` 目录下创建一个全新的独立 Markdown 文件（例如：`20260502-T28-nav-refactor.md`）。
   - 文件内必须详细记录：变更文件列表、核心逻辑改动、设计决策(Trade-offs)、功能自测结果。
   - 这不仅是工作交接依据，更是未来排查 BUG 的救命档案。

### 阶段 3：验证与交付

8. **强制编译与静态检查 (强制红线)**
   - 代码编写完成后，**必须主动调用 run_command 执行** `flutter analyze`。
   - 必须仔细检索日志输出中的 `error` 或 `undefined` 警告。一旦发现编译错误，必须修复至 0 error 状态。
   - **绝对禁止**在存在编译报错的情况下直接生成 Changelog 和更新看板。只有在确保代码通过静态检查后，才能进入后续的归档与交付流程。

9. **同步开发任务列表 (真相源更新)**
   - 打开对应版本的开发任务列表（如 `docs/dev/开发任务清单-v3.0.md`）。
   - **必须**将本次完成的子任务状态手动通过工具替换为 `[x]`。

10. **同步 TaskBoard (真相源更新)**
   - 打开 `docs/TaskBoard.md`。
   - 如果开发清单中该任务编号下的**所有子任务**都已完成，必须将这里标记为 `✅ 已完成`。
   - 如果只完成了部分子任务，保持 `🔄 进行中`。

11. **交付确认**
    - 向用户汇报完成情况，列出：
      - 新增/修改的文件清单
      - 关键设计决策
      - 已知的注意事项
    - 询问是否需要立即进入测试阶段（可触发 `/music-test` 工作流）。

## 断点恢复机制

### 🔴 Checkpoint 恢复保护（最高优先级）

> **背景**：当会话被 Checkpoint 截断恢复时，AI 可能只恢复了"编码目标"的记忆，但丢失了工作流当前执行到哪个步骤的上下文。这会导致编码完成后直接结束，跳过步骤 9（更新开发任务清单）、步骤 10（同步 TaskBoard）、步骤 11（交付确认）等关键收尾动作。

**强制规则**：当 AI 感知到自己处于 Checkpoint 恢复状态时（会话 summary 中提到"truncated"或存在 checkpoint 标记），必须执行以下恢复流程：

1. **重新阅读本工作流文件**（`music-dev-task.md`），刷新完整的步骤清单。
2. **读取 `docs/TaskBoard.md`**，确认当前任务的状态（`🔄` 进行中 / `⬜` 未开始）。
3. **读取 `docs/changes/`** 中最新的变更文件，了解上次做到了哪里，避免重复劳动。
4. **对照阶段 3 的检查清单**，逐项确认状态同步是否被遗漏。
5. **补全所有被遗漏的步骤**，然后再继续新的编码工作。

### 标准断点恢复流程

如果会话中断或开启新会话继续开发（非 Checkpoint 截断）：
1. 读取 `docs/TaskBoard.md`，扫描所有状态为 `🔄`（进行中）和 `⬜`（未开始）的任务。
2. 跳过所有已标记为 `✅` 的完成项。
3. 读取 `docs/changes/` 下最新的几个文件，掌握最近的架构和代码变动。
4. 从断点处继续执行。

## 交付完整性自检清单

> 任务编码完成后，**必须逐项核对**以下清单，全部打勾后方可向用户交付。
> 此清单对应步骤 6~11，任何一项遗漏都视为流程不合格。

```
□ 1. 变更级归档 — 已在 docs/changes/ 生成独立的 markdown 档案文件
□ 2. flutter analyze — 已执行，0 error
□ 3. flutter build web — 已执行，编译通过（如条件允许）
□ 5. 开发任务清单 — 已将完成的子任务标记为 [x]
□ 6. 测试清单 — 已确认对应任务有测试用例覆盖（如无则补充）
□ 7. TaskBoard — 已更新任务状态（全完成 → ✅ / 部分完成 → 🔄）
□ 8. 向用户交付 — 已汇报完成情况（变更文件、设计决策、注意事项）
```

**特别注意**：对于**重构/技术债务类任务**（如 T27），虽然不新增功能，但会影响已有功能的行为。必须在测试清单中补充**回归测试用例**，覆盖被重构模块的核心功能点。

## 应急熔断机制

当自验证阶段发现以下情况时，**必须立即停止**并向用户汇报，禁止继续推进后续步骤：

- `flutter analyze` 报错超过 3 处
- 新增代码导致已有功能异常
- 发现架构设计与 PRD 存在重大偏差
- 涉及 RLS 策略变更但未创建 SQL 文件
- 单文件编辑后超过 300 行
