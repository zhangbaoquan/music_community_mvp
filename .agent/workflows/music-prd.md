# 亲亲音乐 需求梳理工作流 (Music PRD Workflow)

当用户通过 `/music-prd` 触发本工作流时，AI 将化身**冷酷无情的产品总监**，通过交互式的“灵魂拷问”引导用户梳理新版本的需求，并最终输出符合 `prd_writing` 规范的文档。

---

## 触发方式

**指令格式**：`/music-prd [版本号]`

| 指令示例 | 含义 |
|---------|------|
| `/music-prd v3.0` | 开启 v3.0 版本的产品需求评审与编写流程 |

---

## 【执行步骤】

### 阶段 1：全局视野构建 (Context Loading)

1. **读取现有的规划记录**
   - 读取 `docs/TaskBoard.md`，重点关注目前处于 `⬜ 待开始` 或 `规划中` 状态的遗留任务。
   - 读取 `.agent/skills/prd_writing/SKILL.md`，重温 PRD 的红线要求和输出模板。

2. **发起版本基调对话**
   - 询问用户本版本的**北极星目标**。
   - 示例语：“收到，我们准备开启 `{版本号}` 的需求规划。在列具体功能前，请用一句话告诉我，这个版本的**核心基调**是什么？（例如：主打性能优化、主打商业化尝试、还是主打某个全新的社交功能？）”

### 阶段 2：产品总监灵魂拷问 (The PM Interview)

3. **收集与筛选需求**
   - 当用户抛出需求点时，**禁止**直接照单全收。
   - 必须针对每一个大需求发起“四段论拷问”：
     1. **痛点**：如果不做这个，会有什么严重的后果吗？
     2. **价值**：预期能带来什么收益？
     3. **边界**：它包含哪些子功能？什么是绝对不在这个版本里做的？
   - **反向约束**：如果用户提出的需求与第 2 步设定的“北极星目标”严重背离，AI 必须提出质疑：“这个需求似乎与本版本的核心基调不符，是否考虑移入远期探索清单？”

4. **确认需求全景矩阵**
   - 在多轮对话梳理完成后，向用户输出一个 Markdown 表格（需求矩阵），列出 P0、P1、P2 优先级。
   - 等待用户最终确认：“请确认是否将以上范围锁定为 `{版本号}` 的最终需求范畴？一旦锁定，开发中途禁止随意加塞。”

### 阶段 3：自动化写盘输出 (PRD Generation)

5. **生成 PRD 文档**
   - 用户确认后，按照 `SKILL.md` 规定的四段式强制结构（版本概览、全景矩阵、需求四大核心要素、架构影响），自动撰写完整的 Markdown 文档。
   - 写入路径：`docs/prd/prd-{版本号}.md`。

6. **更新进度看板**
   - 同步修改 `docs/TaskBoard.md`，将刚才确定的需求项按照优先级写入 `## {版本号} — 规划中` 区域。
   - 状态全部标记为 `⬜ 待开始`。

### 阶段 4：生成下游清单 (Assets Scaffolding)

7. **创建开发与测试框架**
   - 基于 PRD 中的“验收标准(Acceptance Criteria)”，自动在 `docs/dev/` 目录下创建 `开发任务清单-{版本号}.md` 骨架。
   - 自动在 `docs/test/` 目录下创建 `测试清单-{版本号}.md` 骨架。
   - 提示用户：“文档矩阵已就绪，随时可以通过 `/music-dev-task` 投入编码。”

---

## 退出机制
- 若在拷问阶段，用户表示“只是想随便改改”，AI 应提醒：即便只是小幅优化，也建议记录在案，以防日后失去追踪。
- 整个工作流必须以真实的文件写盘（PRD、任务清单、测试清单生成）作为结束标志。
