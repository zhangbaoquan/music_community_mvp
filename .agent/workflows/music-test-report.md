# 亲亲音乐 测试报告工作流

当用户通过 `/music-test` 触发本工作流时，进入标准测试执行与报告生成模式。工作流会根据指定的测试模块自动执行测试、收集证据、生成规范化报告并归档。

---

## 触发方式

**指令格式**：`/music-test [测试模块编号]`

| 指令示例 | 含义 |
|---------|------|
| `/music-test FC-01` | 执行 FC-01 项目启动与路由 的测试 |
| `/music-test FC-04` | 执行 FC-04 社交互动 的测试 |
| `/music-test FC-09` | 执行 FC-09 资源管道化处理 的测试（v2.0） |

**测试编号路由表**：

| 编号范围 | 所属版本 | 测试清单路径 |
|---------|---------|------------|
| FC-01 ~ FC-08 | v1.0 回归测试 | `docs/test/测试清单-v1.0.md` |
| FC-09 ~ FC-15 | v2.0 新功能测试 | `docs/test/测试清单-v2.0.md` |

**报告存放目录**：`docs/test_reports/`

---

## 【红线规则】

### 🔴 报告格式统一红线

所有测试报告**必须严格遵循以下统一模板**：

**报告头部**：测试日期、测试环境、测试人员、测试范围

**第一节. 测试概览**：汇总表格（模块 | 用例数 | 通过 | 失败 | 通过率）

**第二节. 详细测试记录**：每条用例采用标准 QA 四栏矩阵表格：

| 项目 | 内容 |
|------|------|
| 前置条件 | ... |
| 操作步骤 | ... |
| 预期结果 | ... |
| 实际结果 | ✅ 通过 / ❌ 失败 - 简要描述 |

**第三节. 证据附件**：截图或录屏

**第四节. 发现的问题与改进建议**：问题表格（编号 | 严重级别 | 描述 | 状态）

**第五节. 结论**：一句话总结

### 🔴 证据留存红线

- 每个关键测试步骤**必须**截图保存为 PNG/WebP
- 完整测试流程**应当**录屏保存
- 截图和录屏保存到 `docs/test_reports/media/` 目录下

### 🔴 物理写盘红线

报告文件和相关媒体**必须**写入 `docs/test_reports/` 目录下，禁止仅在对话中展示。

### 🔴 多端适配测试红线（针对前端 UI 功能）

由于基于 Flutter Web 构建，任何涉及前端 UI 呈现和交互响应的测试项目，**必须在两套分辨率下分别进行测试并独立记录结果**：
1. **大屏模式 (PC 端)**：例如 `1440x900`
2. **小屏模式 (Mobile 端)**：例如 `400x800`
测试报告中必须明确体现不同屏幕尺寸下的功能差异。

## 执行步骤

### 阶段 1：测试准备

1. **解析测试目标**
   - 从触发参数中识别测试模块编号（如 FC-04、FC-09）。
   - 根据编号路由到对应版本的测试清单。

2. **加载测试用例**
   - 阅读对应测试清单中的对应章节。
   - 提取所有待测试用例（状态为 `[ ]` 的条目）。
   - 理解每条用例的前置条件、操作步骤和预期结果。

3. **环境检查**
   - 确认 Flutter Web 开发服务器是否运行中（`flutter run -d web-server --web-port=3000`）。
   - 如需要，协助用户启动开发服务器。
   - 确认 Supabase 服务可用（通过简单的 API 调用验证）。

4. **涉及 Realtime 功能的额外检查**
   - 如果测试模块涉及实时功能（评论实时推送、私信等），必须**验证 WebSocket 连接状态**。
   - 检查 Supabase Realtime Channel 是否正常订阅。

### 阶段 2：执行测试

5. **执行测试（多端双视角执行）**
   - 针对涉及 UI 的前端用例，务必控制 `browser_subagent` 调整浏览器视口尺寸（Viewport），分别在**大屏幕 (例如 1440x900)** 和 **小屏幕 (例如 400x800)** 下各跑一遍流程。
   - 对比检查页面排版、弹窗、事件监听是否存在跨端适配的 BUG。
   - 每个关键步骤分别自动截图。
   - 如涉及 Supabase RPC 验证（后端测试），使用 curl 命令调用并记录响应即可。

6. **收集证据**
   - 截图保存到 `docs/test_reports/media/` 目录下。
   - 命名规则：`FC{编号}_{步骤描述}.png`（如 `FC04_评论实时推送.png`）。

### 阶段 3：生成报告

7. **撰写测试报告**
   - 按照统一模板格式生成 Markdown 报告。
   - 命名规则：`FC-{编号}_{模块名}_测试报告.md`
   - 示例：`FC-04_社交互动_测试报告.md`
   - 写入 `docs/test_reports/` 目录。

8. **更新测试清单状态**
   - 将测试清单中对应的条目状态更新：
     - 通过：`[x]` 已通过
     - 失败：`[x]` ❌ 未通过，附注失败原因

### 阶段 4：归档与汇报

9. **交付确认**
   - 向用户展示测试通过率汇总。
   - 如有失败用例，建议是否回退到 `/music-dev` 修复后重新测试。
   - 列出所有发现的问题及改进建议。
