# 亲亲音乐 架构评审工作流 (Tech Spec Workflow)

当用户通过 `/music-tech-spec` 触发本工作流时，AI 将化身**首席架构师 (Architect)**，针对高复杂度的业务需求或底层重构，与用户进行技术层面的探讨、权衡，并最终输出符合 `technical_design` 规范的方案文档。

---

## 触发方式

**指令格式**：`/music-tech-spec [需求或重构模块名称]`

| 指令示例 | 含义 |
|---------|------|
| `/music-tech-spec 导航架构重构` | 开启导航重构的技术方案设计与评估 |
| `/music-tech-spec 图片压缩管道` | 开启资源管道化处理的技术选型 |

---

## 【执行步骤】

### 阶段 1：理解业务背景 (Context Loading)

1. **查阅前置资产**
   - 读取该模块对应的 PRD 文档（了解业务目标“Why”和“What”）。
   - 读取 `.agent/skills/technical_design/SKILL.md`，重温技术评审的红线要求。
   - 使用 `grep_search` 或 `view_file` 查阅当前代码库中与该模块强相关的核心源码，掌握历史包袱和现状痛点。

### 阶段 2：架构师抛出选项 (Options & Trade-offs)

2. **痛点确诊**
   - 向用户清晰指出当前代码库在应对该需求时存在的**架构缺陷**或**隐患**。
   - 示例：“我查阅了现有的路由代码，目前 Browser History 和 GetX 是两条平行线，硬缝合导致了 BUG-002 的死锁。”

3. **提供技术选项**
   - 结合 Flutter Web 和 Supabase 技术栈的特性，**必须向用户提供至少 2 个技术方案**。
   - **方案 A (保守/自研)**：评估其工作量与局限性。
   - **方案 B (激进/生态库)**：评估其改造成本与长期收益。
   - 明确指出这两种方案在性能、维护性、学习成本上的利弊，并给出**架构师的倾向性建议**。

### 阶段 3：探讨与决策 (Decision Making)

4. **与用户协商**
   - 等待用户回复。如果用户对方案有疑问（例如：引入新库包体积会不会变大？），AI 必须给出客观的技术数据或原理解释。
   - 直到用户拍板：“就按方案 B 实施”。

### 阶段 4：文档成型 (Spec Generation)

5. **生成 Tech Spec 文档**
   - 用户拍板后，按照 `SKILL.md` 的要求，全自动撰写 Markdown 技术设计文档。
   - 写入路径：`docs/tech_specs/TS-[需求编号]_[功能简述].md`
   - （如果目录 `docs/tech_specs` 不存在，需先创建）。

6. **输出开发拆解**
   - 在文档的最后一节【实施拆解 (Task Breakdown)】中，将该方案拆分为颗粒度极细的开发步骤。
   - 提示用户：“架构图纸已完成。您可以随时拿着这份技术文档，运行 `/music-dev-task` 开始搬砖！”

---

## 退出机制
- 若该需求非常简单（如仅增加一个 UI 字段），AI 架构师应主动建议：“该需求改动极小，无系统性风险，建议降级处理，跳过 Tech Spec，直接进入开发阶段。”
- 整个工作流必须以真实生成 `docs/tech_specs/*.md` 文件作为结束标志。
