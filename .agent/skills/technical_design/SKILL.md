# 亲亲音乐 技术方案设计规范 (Technical Design Specification)

> “在写下第一行代码前，我们必须清楚知道这行代码在半年后会不会成为新的技术债。”
> 本规范旨在为复杂需求建立**技术评审护栏**，隔离产品需求与技术实现，确保架构演进的科学性。

---

## 1. 适用范围与触发条件

并不是所有需求都需要写 Tech Spec。只有当满足以下任意条件时，**必须**先产出技术方案：
- 涉及跨模块的数据流转或状态管理变更（如全局 Player 与页面路由的联动）。
- 涉及引入第三方重量级依赖（如新的视频播放器、新的状态管理库）。
- 涉及 Supabase 数据表结构的变更或 RLS 策略的重写。
- 历史代码的重构与解耦（如消除 GetX 路由、巨型单文件拆分）。
- **优化类需求**（性能、首屏、防抖节流），没有度量指标和基准线禁止动工。

---

## 2. 强制结构模板

所有 Tech Spec 文档（存放在 `docs/tech_specs/`）必须遵循以下结构：

### 2.1 需求上下文 (Context & Objective)
- **目标**：一句话概括要解决的工程问题（来自 PRD 的哪一项）。
- **非目标 (Non-Goals)**：明确在这个技术方案中**不解决**什么问题，防范过度设计。

### 2.2 当前系统痛点 (Current State & Pain Points)
- 客观描述现状，包括现有代码的缺陷、性能瓶颈、或是导致 BUG 的根本原因。

### 2.3 方案选型与对比 (Trade-offs / Alternatives)
*这是 Tech Spec 最核心的部分。*
必须列出至少 2 个可选方案，并从以下维度进行横向对比：
- **方案 A（通常是造轮子/原生实现）**：优缺点、维护成本。
- **方案 B（通常是引入生态/重构架构）**：优缺点、学习曲线。
- **最终决策 (Decision)**：明确选择哪个方案，并说明决定性因素是什么。

### 2.4 详细设计 (Detailed Design)
针对选定的方案，阐述实施细节：
- **数据流 / 状态管理**：数据如何获取？存储在哪里？如何触发 UI 刷新？
- **接口 / 数据库变更**：涉及到的 Supabase 表结构、RPC 函数、或是新的 API 字段。
- **核心逻辑伪代码/流程**：复杂的算法或时序逻辑。

### 2.5 风险评估与兜底 (Risks & Fallbacks)
- **技术风险**：这个改动最容易在哪里翻车？（例如：破坏了现有的历史数据解析？）
- **兜底策略**：如果有 BUG，如何快速降级？

### 2.6 实施拆解 (Task Breakdown)
- 将庞大的方案拆解为可独立提交（Commit）、可独立测试的细颗粒度任务。
- 直接作为后续生成 `开发任务清单` 的依据。

---

## 3. 协作原则

- **技术中立**：AI 作为架构师，不得因为“某框架流行”而盲目推崇，必须结合当前项目上下文（Flutter Web 侧重、WASM 支持、Supabase BaaS 体系）给出客观评价。
- **文档先行**：技术方案未达成共识并落地为 Markdown 文件前，**绝对禁止修改项目源码**。
