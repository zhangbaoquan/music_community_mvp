## Skill 用途
这个 SkilL用来规范技术文章的撰写流程和输出质量，重点覆盖:
- 技术文章的章节结构和标题层级规范
- 流程图、时序图、架构图等图例的使用规范和文字描述要求
- 代码示例的选取原则和注释标准
- 确保每篇文章有完整的背景分析、设计原理、实战用例和总结，便于团队成员快速理解和使用

# 为什么存在
团队在为核心模块、架构组件撰写技术分析文章时，容易出现风格不统一、结构缺失、图例无说明、代码示例与源码脱节等问题。
这个SkilL把文章结构、图例规范、代码示例要求等统一成稳定做法，确保产出的每篇文章都能成为团队可复用的知识资产。

# 适用范围

- 为项目核心模块、架构组件撰写技术分析文章的场景
- 输入:需要分析的模块/组件名称、所在工程目录
- 输出:一篇符合本规范的`.md`技术文章，包含架构图、流程图、代码示例和总结

## 禁止项

- 禁止图例(流程图，时序图，架构图等)没有配套的文字描述，每张图的上方或下方必须有一段文字说明该图的含义
- 禁止代码示例中出现与项目实际 API不符的调用方式，必须先查阅源码确认正确的方法名和参数签名
- 禁止使用纯英文标题，章节标题必须使用中文，格式为"一、Xxx"、"1.1xXx"、"1、1、1xxx"
- 禁止文章缺少总结章节，最后一章必须是总结(包含组件对照表和注意事项)
- 禁止文章没有总标题，文件首行必须是'# 标题名称'，文件名必须与标题名称一致
- 禁止使用占位示例或虚构的类名/方法名，所有代码示例必须来自项目真实代码或基于真实代码改写

## 执行约束

### 文章结构

每篇技术文章必须包含以下章节(可根据主题适当增减子章节)：

```
# 标题名称
> 一句话简介
# 一、背景            - 为什么需要这个组件/模块?解决什么问题?
# 二、整体架构        - 分层架构图 + 组件关系说明
# 三~N、核心模块详解   - 每个核心类/组件单独一章，包含职责、流程图、代码分析
# N+1、实战用例       - 2~3 个真实场景的完整代码示例
# N+2、总结          - 组件对照表 + 核心理念 + 注意事项
```

### 标题层级

| 层级 | 格式 | 示例 |
|------|
| 一级章节 | # 一、xXx | # 一、背景:为什么需要AceBus |
| 二级章节 | ## 1.1XXx | ## 1.1 问题:多引擎数据隔离 |
| 三级章节 | ### 1.1.1xxx | ### 1.1.1 具体表现 |

### 图例规范
**图例类型选择：**

| 场景 | 推荐图例类型 |
|------|------|
| 组件之间的关系和依赖 | graph TB / graph LR (架构图) |
| 方法内部的执行步骤 | flowchart TD (流程图) |
| 多角色之间的交互过程 | sequenceDiagram (时序图) |
| 决策判断 | graphTD+菱形节点(决策图) |
| 数据对比 | 表格(Markdovn Table) |

**文字描述要求：**每张图例必须满足以下结构：

```
[图例上方] 一段文字描述该图要表达的核心信息，引导读者理解图的含义。
```mermaid
(图例内容)
```(结束符号)
[图例下方] (可选) 补充说明，注意事项或关键节点解释。
```

### 代码示例

- 代码示例必须标注语言类型(kot1in、dart等)
- 关键行必须有注释说明f示例前必须说明场景(如"**Android 端** - 注册监井写入状态")
- 如有双端代码(Android + Flutter)必须成对出现并分别标注
- 所有类名、方法名必须与项目源码一致，写之前必须先查阅源码确认

### 表格使用

以下场景推荐使用表格提升可读性:

- 职责概览(方法名+说明)
- 对比分析(A 方案 VS B 方案)
- 枚举值说明(状态值+含义+UI 表现)
- API 速查
- 注意事项清单

### 文件命名
- 文件名必须与文章总标题一致，使用.md扩展名
- 例如标题为 # AceBus:跨引擎事件总线全解析，文件名为AceBus:跨引擎事件总线全解析.md

**图例 +文字描述模板:**
```markdown
下图展示了 xxx 的完整执行流程。首先 A调用 B，然后 B通过C广播给所有D。
注意 E 环节会引入一次主线程切换的微小延迟。
```mermaid
sequenceDiagram
    participant A as 调用方
    participant B as 中转层
    A->>B:调用方法
    B->>B:处理逻辑
```(结束符号) 
整个链路中的关键节点是 B 的处理逻辑，它决定了消息的分发方式。
```

# 修改注意事项
- 不要产出没有总结章节的文章
- 不要在图例旁边省略文字描述
- 不要使用虚构的类名/方法名作为代码示例
- 不要忘记检查代码示例中的 API 签名是否与项目源码一致
