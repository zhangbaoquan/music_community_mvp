## Skill 用途
这个 Skill 用来规范测试报告的撰写流程和输出质量，重点覆盖：
- 测试报告的统一章节结构和模板格式
- 前端（截图/录屏）和后端（curl/JSON）证据留存标准
- 问题发现与改进建议的记录规范
- 确保每份报告都能作为质量验收的正式交付物

# 为什么存在
在 AI 辅助开发中，测试环节是质量的最后一道防线。如果测试报告格式不统一、证据缺失、问题描述模糊，就无法有效追溯 Bug、复现问题或向团队证明功能已通过验证。
这个 Skill 把测试报告的结构和证据要求固化为标准做法，确保前后端报告质量一致，每份报告都具备可审计性。

# 适用范围

- 在 Luban 项目中完成测试执行后，生成测试报告的场景
- 输入：测试模块编号（如 FC-01、TC-06）、测试执行结果、截图/录屏/API 响应等证据
- 输出：一份符合本规范的 `.md` 测试报告，存放在对应工程的 `docs/test_reports/` 目录下

## 禁止项

- 禁止测试报告缺少报告头部（测试日期、环境、人员、范围），每份报告开头必须包含完整的元信息
- 禁止用例记录缺少四栏矩阵（前置条件、操作步骤、预期结果、实际结果），每条用例必须完整填写
- 禁止前端测试报告没有截图证据，每个关键步骤必须附带 PNG 截图
- 禁止后端测试报告没有 API 响应证据，每个接口测试必须保留完整的 curl 命令和 JSON 响应体
- 禁止问题描述使用模糊表达（如"有点不正常"），必须包含具体的错误现象、复现步骤和严重级别
- 禁止测试报告没有结论章节，最后一节必须有一句话总结测试通过率和整体质量评估
- 禁止仅在对话中展示报告内容而不写入物理磁盘文件

每份测试报告必须包含以下五个章节：

```markdown

> **测试人员**: AI 辅助 + 人工审查
> **测试范围**: [模块编号及名称]

## 第一节、测试概览

### 证据留存标准

| 测试类型 | 必须留存的证据 | 存放位置 |
|---------|-------------|---------|
| 前端测试 | 每个关键步骤的 PNG 截图 + 完整流程的录屏视频 | `docs/test_reports/media/` |
| 后端测试 | 完整的 curl 命令 + HTTP 状态码 + JSON 响应体 | 内嵌在报告的 `<details>` 折叠块中 |

### API 证据的折叠格式

后端 API 测试的原始报文使用以下格式收纳，避免报告过长：

```markdown
<details>
<summary>🔍 [接口名称] 请求/响应详情</summary>

**请求**：
​```bash
curl -X POST http://127.0.0.1:8080/api/xxx -H "Content-Type: application/json" -d '{"key":"value"}'
​```

**响应**（HTTP 200）：
​```json
{"status": "success", "data": {...}}
​```

</details>
```

### 严重级别定义

| 级别 | 含义 | 处理策略 |
|------|------|---------|
| 🔴 P0-致命 | 核心功能不可用，阻塞发布 | 必须立即修复 |
| 🟠 P1-严重 | 主要功能受损，有 Workaround | 当前迭代内修复 |
| 🟡 P2-一般 | 次要功能异常或体验不佳 | 排入下一迭代 |
| 🟢 P3-轻微 | 文案错误、样式微调等 | 有空时修复 |

### 文件命名

- 前端报告：`FC-{编号}_{模块名}_综合测试报告.md`
- 后端报告：`TC-{编号}_{模块名}_综合测试报告.md`
- 示例：`FC-04_组件管理_综合测试报告.md`

# 修改注意事项
- 不要产出没有测试概览汇总表的报告
- 不要省略证据附件，尤其是前端截图和后端 API 响应
- 不要在问题表格中使用模糊的严重级别描述，必须使用 P0~P3 标准
- 写完报告后必须同步更新测试清单/测试用例文档中对应条目的状态
