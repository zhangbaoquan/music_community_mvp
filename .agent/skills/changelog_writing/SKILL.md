## Skill 用途
这个 Skill 用来规范变更日志（Changelog）的撰写流程和输出质量，重点覆盖：
- 变更日志的条目结构和必填字段规范
- 变更文件列表的精确性要求
- 设计决策的记录标准（何时必须写、写什么）
- 确保每条日志能支撑"断点恢复"——新会话通过读取 Changelog 尾部即可了解上次做到哪里

# 为什么存在
AI 在跨会话开发中最大的痛点是"失忆"。如果每次代码变更后不留下精确的变更记录，下次对话将完全不知道上次做了什么、改了哪些文件、为什么做出某个技术选择。
这个 Skill 把变更日志的结构和内容要求固化为稳定做法，确保每条记录都能成为跨会话断点恢复的可靠锚点。

# 适用范围

- 在 Luban 项目中完成一组有完整意义的代码变更后，追加变更日志的场景
- 输入：本次变更涉及的任务编号、修改的文件列表、变更内容描述
- 输出：一条符合本规范的 Changelog 条目，追加到对应工程的 `docs/Changelog.md`

## 禁止项

- 禁止变更日志中使用模糊描述，如"修改了一些文件"、"优化了代码"，必须指明具体文件路径和变更内容
- 禁止遗漏变更文件，所有新增、修改、删除的文件路径必须逐一列出
- 禁止省略设计决策，当存在非显而易见的技术选型或取舍时（如"选择 WebSocket 而非轮询"），必须记录理由
- 禁止将多个不相关的任务合并到同一条变更记录中，每个任务编号独立成条
- 禁止日期格式不统一，统一使用 `YYYY-MM-DD` 格式
- 禁止在变更日志中出现与实际代码不一致的文件路径或类名

## 执行约束

### 条目结构

每条变更日志必须包含以下字段：

```markdown
## [YYYY-MM-DD] [任务编号] — [一句话简述]
- **变更文件**：
  - [新增] `path/to/new_file.dart`
  - [修改] `path/to/modified_file.dart`
  - [删除] `path/to/deleted_file.dart`
- **变更内容**：一句话说明做了什么（需具体到功能点）
- **设计决策**：（可选，当存在非显而易见的选型或取舍时必填）简要说明为什么这样做
```

### 变更文件标记

| 标记 | 含义 |
|------|------|
| `[新增]` | 本次新创建的文件 |
| `[修改]` | 本次修改了已有文件 |
| `[删除]` | 本次删除了已有文件 |

### 设计决策的触发条件

以下场景**必须**记录设计决策：
- 在多个技术方案中做了选择（如 riverpod vs provider）
- 引入了新的第三方依赖
- 修改了已有的 API 接口签名或数据库表结构
- 采用了与 PRD 描述不完全一致的实现方式（需说明原因）
- 做了性能或安全相关的权衡取舍

### 追加位置

- 新条目始终追加在 `Changelog.md` 文件的**最顶部**（最新的在最上面）
- 不要修改或删除已有的历史条目

# 修改注意事项
- 不要产出没有具体文件路径的变更记录
- 不要把设计决策漏掉，尤其是架构级别的选型
- 不要在一条记录里混入多个任务的变更
- 写完后检查文件路径是否与工程中的实际路径一致
