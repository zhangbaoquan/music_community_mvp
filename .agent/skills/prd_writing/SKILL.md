# 亲亲音乐 PRD 编写规范与需求约束机制

> 独立开发者最大的死敌是“想到什么加什么”。
> 本规范旨在通过强制的结构化输出和“灵魂拷问”机制，确保每一个进入开发队列的需求都有清晰的价值和明确的边界。

---

## 1. 核心约束原则（产品评审标准）

任何被正式写入 PRD 的新功能，**必须**能够回答以下三个问题：
1. **背景痛点**：为什么要做？如果不做会怎样？
2. **预期收益**：解决什么问题？带来了什么可衡量的价值？（是日活提升、留存提升，还是防崩溃？）
3. **优先级与版本契合度**：它符合当前版本的“北极星目标”吗？能不能放到下个版本？

如果一个需求无法清晰回答上述问题，必须将其打回或放入 `远期探索(Backlog)`。

---

## 2. PRD 强制结构模板

新版本的 PRD（如 `docs/prd/prd-v3.0.md`）必须严格按照以下四段式结构组织：

### 2.1 版本概览（The Big Picture）
- **版本号**：例如 v3.0
- **版本北极星目标（核心基调）**：一句话定义当前版本。例如：“v2.0 专注偿还技术债务”，“v3.0 专注交互性能重构与架构升级”。
- **目标用户体验**：交付给用户最直观的改变是什么。

### 2.2 需求全景矩阵（Feature Matrix）
用一个表格概览本版本的所有核心需求，必须包含：
- **编号**：如 F-01
- **模块/功能名称**：如“全局缓存管理”
- **优先级**：P0(阻断性/核心)、P1(高价值)、P2(锦上添花)
- **状态**：规划中/已上线

### 2.3 需求详细描述（四大核心要素）
这是 PRD 的灵魂部分，每个需求（F-01, F-02...）必须包含这四个子块：

#### F-XX [功能名称]
- **【背景与痛点】**：现有设计有什么问题？（例如：用户每次进入都要重新加载十几兆的图片，消耗流量且卡顿）。
- **【核心价值与预期收益】**：我们期望达到什么效果？（例如：实现秒开，大幅提升列表滚动流畅度）。
- **【业务逻辑与规则】**：
  - 核心功能点 1 的规则描述
  - 核心功能点 2 的规则描述
  - 异常流/边界状态（如：断网怎么处理？数据空怎么处理？）
- **【验收标准 (Acceptance Criteria)】**：
  - 必须是一系列可执行、可验证的条目（将直接转化为下一阶段的测试清单）。
  - 例：[ ] 无网络状态下能展示缓存数据。

### 2.4 数据与架构影响（可选但强建议）
- 该需求是否需要修改现有的 Supabase 表结构或 RLS 策略？
- 该需求是否引入了新的重量级第三方依赖？

---

## 3. 需求变更（加塞）原则

在版本开发期间（特别是已经进入编码阶段后）：
- **禁止无痕加塞**：如果 Owner 突然产生新的灵感，**绝对禁止**直接把新功能写进代码。
- **强制阻断**：AI 必须提醒 Owner：“这是一个新需求”。
- **降级处理**：优先将新灵感记录到《远期探索清单》中，留待下个大版本讨论；如果极度重要必须插入当前版本，必须走变更流程：重新更新 PRD -> 重新评估开发周期。
