## 3. 需求详细描述

### F-01 [T28 导航架构深度重构]
- **【背景与痛点】**：现有的 GetX 路由拦截与手动封装的 Browser History API 存在强耦合与冲突。用户在某些场景下点击浏览器的“后退”按钮会导致页面死锁，且部分子页面的 URL 无法准确直达。
- **【核心价值与预期收益】**：消除架构隐患，实现 URL 即状态。让每一次分享都精准无误。
- **【业务逻辑与规则】**：
  - 彻底解耦 UI 状态切换与 URL 变化。
  - 所有带有唯一 ID 的实体（如文章详情、他人主页、特定歌曲播放），必须映射为唯一的 URL 路径。
- **【验收标准 (Acceptance Criteria)】**：
  - [ ] 复制浏览器地址栏的任意一首歌/一篇文章的 URL 并在无痕模式打开，能够完美还原该页面。
  - [ ] 在 App 内连续跳转 5 个深层页面后，高频点击浏览器的“后退”和“前进”键，页面不发生死锁或白屏。

### F-02 [T19 资源管道化处理]
- **【背景与痛点】**：用户上传原生高清图片（动辄数 MB），导致云存储成本飙升，同时在弱网环境下，信息流加载极其缓慢，体验极速下降。
- **【核心价值与预期收益】**：既为服务器节省大量的流量与存储费用，又大幅缩减终端用户的 FCP（首屏内容绘制）和 LCP 耗时。
- **【业务逻辑与规则】**：
  - 用户选定媒体文件后，执行压缩策略。
  - 需保证压缩后的视觉质量处于“可接受”范围（无明显马赛克）。
- **【验收标准】**：
  - [ ] 上传一张大于 5MB 的图片，抓包确认实际上传的 payload 小于 500KB。
  - [ ] 压缩后的图片在 1080P 显示器上查看无明显画质裂化。

### F-03 [T20 & T21 数据流重构：游标分页与 RPC 聚合]
- **【背景与痛点】**：当前列表采用全量拉取策略，且大量聚合计算（如点赞数、评论数）依赖客户端内存 Map 遍历。当数据量膨胀时，必定导致浏览器内存溢出（OOM）和严重的卡顿。
- **【核心价值与预期收益】**：突破单页数据量天花板，将重计算压力转移至数据库层，实现客户端的极致轻量化。
- **【业务逻辑与规则】**：
  - 心情电台（Feed 流）、个人中心列表全部接入分页加载。
  - 复杂的关联计数统计必须通过 PostgreSQL 的 RPC 并在服务端缓存/返回。
- **【验收标准】**：
  - [ ] 压测场景：单用户拥有 10,000 条日记时，访问个人主页依然秒开，且滚动到底部时能平滑加载下一页。
  - [ ] Network 面板确认：不再有全量数据拉取的请求，单次请求数据量被严格限制（如 20 条/页）。

### F-04 [T22 Safari 音视频兼容性修复]
- **【背景与痛点】**：苹果 Safari 浏览器对媒体自动播放和跨域资源有着最严苛的安全策略，导致部分音频在 Safari 环境下无法正常播放。
- **【核心价值与预期收益】**：补齐苹果生态用户的核心体验，实现真正的全平台一致性。
- **【验收标准】**：
  - [ ] Mac/iOS 的 Safari 浏览器下，能够正常播放音乐金库中的歌曲。
  - [ ] 点击播放时的交互不被浏览器的 Auto-play 策略静音拦截。

## 4. 数据与架构影响

- **重大架构变更**：F-01（导航重构）和 F-03（游标分页）属于极高风险操作，将直接重构全局 `NavigationController` 以及全部列表 View 组件。
- **技术规范执行**：根据团队规范，**F-01、F-02 和 F-03 在进入开发前，必须先经过 `/music-tech-spec` 进行技术方案选型与评审**。
