## 1. 产品概述

### 1.1 产品定位

亲亲心情笔记是一个主打 **"音乐 + 心情日记"** 的纯粹内容社区。用户在记录此刻心情的同时，可以自由搭配一首能够代表当下的原创或喜爱的心境音乐，实现双向情感表达。

### 1.2 核心设计理念

- **极简认知，降噪表达**：主色调采用克制的黑（`#1A1A1A`）白灰体系，大面积留白，让用户聚焦于文字与音乐
- **沉浸式背景伴随**：全局持久化音乐播放引擎，任何页面音乐无缝流转、持续相伴
- **正向激励引擎**：轻量级游戏化设计（成就徽章系统），冷启动初期给早期创作者正向反馈

### 1.3 目标用户

- 有日记/心情记录习惯的年轻用户
- 喜欢用音乐表达情绪的内容创作者
- 追求纯粹、私密表达空间的社区用户

### 1.4 技术选型

| 技术 | 选型理由 |
|------|---------|
| Flutter Web | 跨端复用，未来可直接出 iOS/Android 包 |
| GetX | 轻量状态管理 + 路由 + DI，适合快速迭代 |
| Supabase | 开箱即用的 Auth/DB/Realtime/Storage，独立开发者友好 |

## 3. 数据模型

### 3.1 核心表结构

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| `profiles` | 用户元数据 | id, username, avatar_url, bio, role, banned_until |
| `articles` | 文章/日记 | id, user_id, title, content, cover_image_url, song_id, type |
| `songs` | 音乐资产 | id, user_id, title, audio_url, duration |
| `comments` | 层级评论 | id, article_id, user_id, content, parent_id |
| `follows` | 关注图谱 | follower_id, following_id |
| `article_likes` | 文章点赞 | user_id, article_id |
| `collections` | 文章收藏 | user_id, article_id |
| `profile_visits` | 访客流水 | visitor_id, profile_id |
| `badges` | 成就徽章 | id, user_id, badge_type, earned_at |
| `private_messages` | 私信 | id, sender_id, receiver_id, content |

### 3.2 安全策略

- 所有表均配置 Row Level Security (RLS) 策略
- 用户只能修改/删除自己的数据
- 管理员通过 role 字段获得额外权限
- Storage Bucket 实行读写物理鉴权隔离

## 4. 非功能需求

### 4.1 性能

| 指标 | 要求 | 实现方式 |
|------|------|---------|
| 首屏加载 | < 3s | CSS 骨架动画 + SW 移除 + 字体优化 |
| 音频连续性 | 页面切换不中断 | GetX 单例 Controller |
| 静态资源缓存 | 30 天强缓存 | Nginx expires 配置 |

### 4.2 安全

| 指标 | 实现方式 |
|------|---------|
| API 鉴权 | Supabase RLS + JWT |
| 流量防护 | Nginx limit_req_zone + limit_conn_zone |
| 爬虫拦截 | User-Agent 黑名单 |
| HTTPS | 腾讯云 CDN 前置 SSL 卸载 |

### 4.3 部署

| 环节 | 方案 |
|------|------|
| 构建 | `flutter build web --release --pwa-strategy=none` |
| 推送 | rsync 差量同步到生产服务器 |
| CDN | 腾讯云 CDN 全国节点加速 |
| SSL | TrustAsia (DigiCert) 免费通配证书 |

## 5. 已知限制

| 编号 | 问题 | 影响 | 优先级 |
|------|------|------|--------|
| L01 | Flutter Web 无原生 DOM，SEO 缺失 | 搜索引擎无法抓取内容 | P1 |
| L02 | 富媒体资源未前端压缩 | 存储成本高 + 加载慢 | P0 |
| L03 | 列表无分页控制 | 数据量增大后内存暴涨 | P0 |
| L04 | 前端聚合计算逻辑堆叠 | 复杂度高 + 存在 .count() 类隐患 | P1 |
| L05 | 部分巨型文件 > 25KB | AI 编辑困难 + 维护性差 | P2 |
| L06 | data 层过于单薄 | Supabase 调用散落在 View/Controller 层 | P2 |
| L07 | 测试覆盖率 ≈ 0% | 重构风险高 | P2 |
