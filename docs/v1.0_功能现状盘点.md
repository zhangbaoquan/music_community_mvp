## 🧩 逐模块功能盘点

### 模块 1：项目启动与路由 (`main.dart` + `layout/`)

**实现功能**：
- Supabase 初始化（URL + anonKey + 防缓存 Headers）
- `AppStartupScreen` 闪屏页 → 1.5s 后跳转 `/home`
- JS interop `removeLoading` 销毁 CSS Loading 动画
- `GetMaterialApp` + 命名路由（13 条路由）
- Sentry 错误监控集成
- 国际化配置（zh_CN + flutter_quill 本地化）
- Material Design 3 主题 + Outfit 字体
- `MainLayout`：响应式双列 / 底栏切换（600px 断点）

**潜在风险**：
- ⚠️ `AppStartupScreen`：硬编码 `Future.delayed(1500ms)` 作为初始化等待，没有真正的服务就绪检查。**如果网络慢，1.5s 内 Supabase 还没完成 auth 恢复，会导致用户状态判断异常**。
- ⚠️ **首屏渲染耗时长** — 用户反馈的已知 BUG。可能原因：Flutter Web WASM/CanvasKit 初始化 + 谷歌字体远程加载 + Supabase SDK 初始化三重叠加。
- ⚠️ `main_layout.dart` 441 行，严重超限。
- ⚠️ `MainLayout` 中 `Get.put(MessageController())` 重复调用了 3 次（L98, L192, L351），可能导致实例冲突。
- ⚠️ `BottomNavigationBar` 的 `onTap` 中 index 0 和 1 没有 `navCtrl.changePage()` 调用（L329-336），点击"驿站"和"日记"可能不响应。

---

### 模块 2：用户认证 (`auth/`)

**实现功能**：
- `AuthController`：全局单例，监听 `onAuthStateChange`
- 登录（Email + Password）、注册（含 username metadata）、登出
- 密码重置邮件发送 + URL Token 密码更新
- `UpdatePasswordDialog` 弹窗
- 深链接处理（URL fragment 解析）
- 登录态持久化恢复

**潜在风险**：
- ⚠️ `AuthController` 直接调用 `_supabase.auth` — 虽然 Auth 在 Service 层是否需要封装有争议，但与项目规范不完全一致。
- ⚠️ `_handleAuthRedirect` 使用 `Future.microtask` 做路由跳转，在 Web SPA 下可能与 GetX 路由栈存在竞争条件。
- ⚠️ 注册成功后的提示文案写了 "请登录"，但 Supabase 默认注册后不需要邮箱验证可能直接登录，行为不一致。

---

### 模块 3：首页信息流 (`home/`)

**实现功能**：
- `HomeView`：Tab 切换"心情广场"和"专栏文章"
- `MoodStationView`：4 个心情卡片（开心/忧郁/平静/专注）+ 按心情筛选日记 + 自动播放对应氛围音乐
- `ContentTabView`：TabBar + TabBarView 双 Tab
- 响应式布局（800px 断点）

**潜在风险**：
- ⚠️ `MoodStationView` 中 `Get.put(DiaryController())` 和 `Get.put(MusicController())` 在 `build()` 方法中调用，每次重建都会触发，可能导致不必要的数据重新加载。
- ⚠️ `DiaryView` 被硬编码了 `SizedBox(height: 600)`，在不同屏幕高度下日记列表可能截断或浪费空间。

---

### 模块 4：内容引擎 (`content/`)

**实现功能**：
- `ArticleController`：771 行**上帝 Controller**，涵盖：
  - 文章列表加载（含 JOIN profiles/songs/likes/collections/comments 计数）
  - 单篇文章详情获取（深链接）
  - 点赞/取消点赞（乐观更新 + 通知）
  - 收藏/取消收藏
  - 评论加载（树状评论构建 + fallback 机制）
  - 发布评论 + 评论点赞
  - 文章发布 + 更新 + 删除
  - 封面图上传（Supabase Storage）
- `ArticleEditorView`：flutter_quill 富文本编辑器
- `ArticleDetailView`：文章详情 + 评论抽屉
- `ArticleListView`：专栏文章列表
- `UserArticlesView`：用户文章管理
- `MusicPickerSheet`：音乐绑定选择器
- `ArticleImageEmbedBuilder`：图片嵌入渲染

**潜在风险**：
- 🔴 `article_controller.dart` 771 行 — 严重超限，必须拆分（评论逻辑至少应独立）
- 🔴 **全部 Supabase 调用直接在 Controller 中**，无 Service 层封装
- ⚠️ `fetchArticles()` 一次性加载所有已发布文章（无分页），数据量大时内存爆炸
- ⚠️ `fetchComments()` 中有大量 `print('[DEBUG ...]')` 调试语句残留
- ⚠️ 封面图上传时未指定 `contentType`（L658），违反 Storage 规范
- ⚠️ `article_detail_view.dart` 39KB — **AI 编辑此文件几乎不可能不丢代码**
- ⚠️ `BadgeService` 在 `addComment` 和 `publishArticle` 中通过 `Get.put()` 创建新实例而非 `Get.find()`

### 模块 5：心情日记 (`diary/`)

**实现功能**：
- `DiaryController`：从 `mood_diaries` 表 CRUD
- `DiaryView`：日记列表 + 发布输入框
- 心情筛选（与首页心情卡片联动）
- `DiaryEntry` 模型（内联在 Controller 文件中，未放 `data/models/`）

**潜在风险**：
- ⚠️ 直接调 Supabase，无 Service 层
- ⚠️ `DiaryEntry` 模型与 Controller 耦合在同一文件（违反架构分层）
- ⚠️ `fetchEntries()` 仅查询当前用户的日记，非登录用户打开页面时 `userId == null` 直接 return，**页面可能显示空白无提示**
- ⚠️ 错误提示使用英文（"Error", "Failed to load diaries"），不一致

---

### 模块 6：全局音乐播放器 (`player/`)

**实现功能**：
- `PlayerController`：GetX 单例，跨页面播放不中断
- `just_audio` 引擎：播放/暂停/停止/音量/进度/拖拽
- `PlayerBar`：底部迷你播放条
- `PlayerView`：展开播放面板
- URL 安全转换（HTTP → HTTPS via `toSecureUrl()`）
- 自动播放策略兼容（手势触发提示）

**潜在风险**：
- ⚠️ `PlayerController` 中 `Get.put(LogService())` 可能与 `AppBinding` 中已 `Get.put` 的实例冲突
- ⚠️ 播放完成后 `_player.seek(Duration.zero) + _player.pause()` 的顺序可能导致短暂音频闪烁
- ⚠️ `togglePlay()` 每次暂停都弹 snackbar "已暂停"，体验不佳（高频操作不应有 toast）

---

### 模块 7：音乐金库 (`music/`)

**实现功能**：
- `MusicController`：音乐上传/删除/发现/用户音乐列表
- `UploadMusicView`：音频上传（MP3选择 + 封面选择 + 心情标签）
- 心情电台播放匹配（多关键词模糊匹配 mood_tags）
- Supabase Storage 音频/封面上传

**潜在风险**：
- ⚠️ 直接调 Supabase，无 Service 层
- ⚠️ `fetchSongs()` 一次性 limit(100)，无分页
- ⚠️ 音频上传未指定 `contentType`（L141），违反 Storage 规范
- ⚠️ `uploadProgress` 变量存在但从未真正更新进度值

### 模块 8：社交互动评论 (`social/`)

**实现功能**：
- `CommentsController`：独立的评论系统（与 `content/` 下的评论逻辑可能有重复）
- `CommentsSheet`：评论底部弹出面板
- `CommentService`：**该模块有独立的 Service 层封装**（项目中唯一的业务 Service）
- Supabase Realtime 评论实时推送
- 层级评论（parent_id 树）
- 评论点赞
- `StoryEditorView`：24KB，日记发布编辑器（内含富文本）

**潜在风险**：
- ⚠️ `story_editor_view.dart` 24KB 严重超限
- ⚠️ `comments_sheet.dart` 18KB 超限
- ⚠️ `social/` 和 `content/article_controller.dart` 中**评论逻辑存在重复实现**，可能导致数据不一致
- ⚠️ Realtime Channel 订阅的取消逻辑需确认是否在 `onClose()` 中正确释放

### 模块 9：个人中心 (`profile/`)

**实现功能**：
- `ProfileController`：573 行，涵盖：
  - 个人资料加载/编辑（用户名、签名、头像）
  - 社交统计（粉丝/关注/访客数 — 客户端计数）
  - 被动统计（获赞/获评论/获收藏 — 通过 RPC `get_user_article_stats`）
  - 关注/取消关注
  - 访客记录（upsert）
  - 收藏文章列表
  - 徽章加载
  - 封禁状态检查
  - 游客模式提示
- `ProfileView`：43KB — **项目中最大的文件**
- `UserProfileView`：他人主页
- `EditProfileDialog`：资料编辑弹窗
- `FollowListView`：粉丝/关注列表
- `VisitorListView`：访客列表

**潜在风险**：
- 🔴 `profile_view.dart` 43KB — **必须优先拆分**
- 🔴 `ProfileController` 573 行，承载过多职责
- ⚠️ `fetchUserStats()` 通过 3 次独立 select + `.length` 客户端计数，效率差（应使用 RPC）
- ⚠️ 头像上传未指定 `contentType`（L404）
- ⚠️ `getPublicProfile()` 发起 4 次独立 Supabase 查询获取一个用户的公开信息

### 模块 10：成就徽章 (`gamification/`)

**实现功能**：
- `BadgeService`：徽章配置加载 + 里程碑检查 + 自动发放
- `BadgeDetailView`：徽章展示墙
- `BadgePopup`：获得新徽章的庆祝弹窗（含 confetti 动画）
- `PremiumBadgeWidget`：高级徽章展示组件
- 支持的里程碑类型：`article_count`、`comment_count`、`follower_count`

**潜在风险**：
- ⚠️ `checkCommentMilestones()` 查询的表名是 `comments` 而非 `article_comments`，可能找不到数据
- ⚠️ `checkLikeMilestones()` 方法体是空的（只有注释），功能未实现
- ⚠️ 直接调 Supabase，无 Service 层隔离

### 模块 11：消息与私信 (`messages/`)

**实现功能**：
- `MessageController`：私信 CRUD + 对话列表 + Realtime 订阅
- `ChatDetailView`：聊天详情页（含 Emoji 选择器）
- `ChatListView`：对话列表
- `MessageCenterView`：消息中心（通知 + 私信合并入口）
- Supabase Realtime 新消息推送
- 已读/未读状态管理

**潜在风险**：
- ⚠️ 直接调 Supabase，无 Service 层
- ⚠️ Realtime Channel 在 `clearMessages()` 中通过 `_supabase.removeChannel(_supabase.channel('public:messages'))` 取消，但 `_supabase.channel()` 会创建新实例而非引用已有实例，可能无法正确取消订阅
- ⚠️ `fetchConversations()` 客户端侧分组（无独立 conversations 表），性能 O(n) 随消息量线性增长
- ⚠️ 消息发送后整个线程 `fetchMessages()` 全量刷新，而非追加新消息

### 模块 12：通知系统 (`notifications/`)

**实现功能**：
- `NotificationService`：通知获取 + 已读标记 + 全部已读
- `NotificationModel`：通知数据模型（内联在 Service 文件中）
- `NotificationView`：通知列表展示
- 通知类型：follow、like_article、comment_article、like_comment、feedback_reply、system_warning
- 客户端去重逻辑

**潜在风险**：
- ⚠️ `NotificationModel` 定义在 Service 文件中而非 `data/models/`
- ⚠️ 无 Realtime 订阅，用户需手动刷新才能看到新通知
- ⚠️ 通知列表仅 limit(50)，无分页

### 模块 13：搜索 (`search/`)

**实现功能**：
- `SearchController`：搜索用户/日记/文章
- `SearchView`：搜索界面（输入框 + 分类结果展示）
- 模糊搜索（`ilike '%query%'`）

**潜在风险**：
- ⚠️ 直接调 Supabase，无 Service 层
- ⚠️ 三个搜索查询串行执行，无并行优化
- ⚠️ `_debounceWorker` 声明了但未实际初始化使用（防抖逻辑缺失）
- ⚠️ 文章搜索只搜 title 不搜 content（注释写了 `.or()` 但实际只用 `.ilike('title')`）

### 模块 14：管理后台 (`admin/`)

**实现功能**：
- `AdminController`：406 行，涵盖用户管理、内容审核、举报处理、反馈管理
- `AdminGuard`：路由中间件（权限拦截）
- `AdminLayout`：后台布局（侧栏 + 内容区）
- 管理页面：用户管理、文章管理、评论管理、日记管理、音乐管理、反馈管理、举报管理
- 封禁/解封、内容清空、密码重置（RPC）
- 举报处理流程（删除/隐藏/忽略 + 系统通知违规者）

**潜在风险**：
- ⚠️ `admin_controller.dart` 406 行超限
- ⚠️ 全部直接调 Supabase，无 Service 层
- ⚠️ `clearUserContent()` 中删除的表名 `diaries` 和 `comments`，但实际项目表名是 `mood_diaries` 和 `article_comments`，**可能完全无效**
