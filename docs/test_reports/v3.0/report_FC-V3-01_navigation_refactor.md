## 2. 详细测试记录

### TC-V3-01-01 Web端深层链接直达 (Deep Linking)
**目的：验证用户分享深层链接后，他人直接在浏览器地址栏打开能否无损还原页面。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 服务端运行正常，用户处于未登录或已登录状态。 |
| 操作步骤 | 1. 在浏览器 A 中进入某个他人主页（如 `/profile/xxxx`）或某篇文章详情页（如 `/article/xxxx`）。<br>2. 复制当前浏览器的 URL。<br>3. 打开一个全新的无痕浏览器窗口，粘贴并直接访问该 URL。 |
| 预期结果 | 1. 页面应当直接渲染目标详情页或主页，绝不能强制跳转到首页。<br>2. 底栏的 Tab 状态（若属于主界面）应当精准匹配。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-02 浏览器前进/后退栈防死锁 (History API Resilience)
**目的：验证原 GetX 的“后退死锁”顽疾是否已被根除。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 用户已登录。 |
| 操作步骤 | 1. 在首页点击动态进入【详情页】。<br>2. 在详情页点击头像进入【个人主页】。<br>3. 在个人主页点击粉丝列表进入【关注页】。<br>4. 高频次点击浏览器的“后退”键 3 次。<br>5. 再点击浏览器的“前进”键 2 次。 |
| 预期结果 | 页面的进退必须和浏览器的 URL 瞬间同步，不能出现任何白屏、堆栈错乱或点击无反应的死锁现象。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-03 嵌套路由 Tab 状态持久化 (Stateful Retention)
**目的：验证基于 StatefulShellRoute 的底栏切换是否能够保持子页面的滚动和业务状态。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 应用处于首页（Tab 0）。 |
| 操作步骤 | 1. 在【首页】向下滚动一段距离。<br>2. 点击底栏的【个人中心】（Tab 2）。<br>3. 在个人中心执行某个弹窗操作后关闭。<br>4. 点击底栏重新切回【首页】（Tab 0）。 |
| 预期结果 | 首页必须维持在刚才滚动的具体位置，且不会触发重新拉取 API 的全量刷新（平滑切换）。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-04 全局弹窗上下文兜底 (Context Recovery)
**目的：验证紧急修复的 Get.key 兜底方案是否在所有非路由场景中生效。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 在任意 Tab 页面。 |
| 操作步骤 | 1. 在侧边栏点击【赞助支持】，验证 `Get.dialog` 是否弹出。<br>2. 点击【退出登录】，验证是否弹出确认框。<br>3. 在某个没有 Context 的 GetX Controller 中触发 `Get.snackbar`（如点赞失败/评论成功）。 |
| 预期结果 | 1. 所有 Dialog 和 BottomSheet 必须正常阻断弹出。<br>2. 所有 Snackbar 必须在顶部正常滑出，没有任何静默报错。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-05 路由传参正确性 (Extra Arguments)
**目的：验证重构脚本将 `arguments` 转换为 `extra` 后的强类型传参是否正确。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 存在有数据的动态列表。 |
| 操作步骤 | 1. 点击一条动态卡片，触发传入 Article 对象的路由跳转（`appRouter.push('/article/id', extra: article)`）。<br>2. 观察详情页的初始渲染。 |
| 预期结果 | 详情页应当瞬间渲染出传递的 Article 数据（而非从网络重新加载呈现骨架屏）。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-06 跨模块复杂跳转链路 (Cross-module Navigation Flow)
**目的：验证多个模块串联跳转时的状态同步与参数传递。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 存在带有评论的系统通知。 |
| 操作步骤 | 1. 从【系统通知】点击带有评论的文章，触发带参数跳转（`?autoOpen=true`）。<br>2. 在【文章详情页】自动展开的评论区中，点击某评论者的头像进入【他人主页】。<br>3. 在【他人主页】点击“私信”按钮，进入【聊天详情页】。<br>4. 观察 URL 变化，并尝试点击返回按钮依次退回。 |
| 预期结果 | 1. 每次跳转 URL 必须实时更新到根节点（如 `/article/xxx` -> `/profile/xxx` -> `/chat/xxx`），参数不能带无意义的后缀（如 `?autoOpen=false`）。<br>2. 按步骤返回时栈内存活良好，不出现状态错乱。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-07 响应式屏幕适配导航 (Responsive Design Navigation)
**目的：验证大小屏幕自适应切换时，Shell路由与局部布局的兼容性。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 在桌面端浏览器进行测试。 |
| 操作步骤 | 1. 在宽屏（PC模式）下，侧边导航栏固定，点击不同路由进入子页面。<br>2. 将浏览器窗口逐渐缩小至窄屏（Mobile模式），侧边栏变为底部导航栏。<br>3. 在窄屏下执行页面跳转（如进入文章详情页）。<br>4. 再将屏幕拉大至宽屏模式。 |
| 预期结果 | 页面不能出现白屏或溢出错误，导航控件平滑切换，路由状态在尺寸变化期间保持一致。 |
| 实际结果 | ✅ 通过 |

### TC-V3-01-08 后台管理子页面导航兜底 (Admin Panel Navigation)
**目的：验证独立于 MainLayout 之外的 AdminLayout 子级路由（由于未注册进 ShellRoute 导致的问题）是否被成功推送到根导航器。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 用户具有 Admin 权限。 |
| 操作步骤 | 1. 进入【后台管理】->【用户管理】。<br>2. 点击某条数据，进入【用户详情页】（原为 `Get.to`）。<br>3. 在用户详情页中点击【日志】进入日志分析页。<br>4. 执行弹窗操作（如删除/封禁）。 |
| 预期结果 | 1. 所有子页面必须通过 `rootNavigator` 铺满全屏，不能被困在 Admin 侧边栏内部的局部容器中。<br>2. 对话框（如封禁确认框）正常显示并成功关闭。 |
| 实际结果 | ✅ 通过 |

## 4. 发现的问题与改进建议

| 编号 | 严重级别 | 描述 | 状态 |

| BUG-01 | 高 | URL 未随命令式 API (push) 同步更新，因为 go_router 的 `optionURLReflectsImperativeAPIs` 未开启。 | ✅ 已修复 (`main.dart` 补丁) |
| BUG-02 | 高 | `GetMaterialApp.router` 导致双 Navigator 冲突，拦截了 `rootNavigatorKey`。 | ✅ 已修复 (启用 rootNavigator 强制推屏) |

| BUG-04 | 中 | Admin 管理后台与部分弹窗中漏掉了大量 `Get.to` 到 `appRouter.push` 的转换。 | ✅ 已修复 (9 处全部清理完毕) |

## 5. 结论

**测试结论：通过**

经过两轮排查与深度修复，因替换为 `go_router` 产生的核心架构冲突、Context 不一致、双 Navigator 抢占以及脚本未覆盖的 `Get.to` 遗留问题均已彻底解决。在复杂的 Web 环境（大小屏切换、深层跳转、URL 分发）中，导航表现已达到生产级的顺滑与稳定要求，v3.0 的基石（架构重构）已稳固。可以进入后续的性能与功能研发阶段。
