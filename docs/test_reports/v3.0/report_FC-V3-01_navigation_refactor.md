## 2. 详细测试记录

### TC-V3-01-01 Web端深层链接直达 (Deep Linking)
**目的：验证用户分享深层链接后，他人直接在浏览器地址栏打开能否无损还原页面。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 服务端运行正常，用户处于未登录或已登录状态。 |
| 操作步骤 | 1. 在浏览器 A 中进入某个他人主页（如 `/profile/xxxx`）或某篇文章详情页（如 `/article/xxxx`）。<br>2. 复制当前浏览器的 URL。<br>3. 打开一个全新的无痕浏览器窗口，粘贴并直接访问该 URL。 |
| 预期结果 | 1. 页面应当直接渲染目标详情页或主页，绝不能强制跳转到首页。<br>2. 底栏的 Tab 状态（若属于主界面）应当精准匹配。 |
| 实际结果 | ⬜ 待测试 |

### TC-V3-01-02 浏览器前进/后退栈防死锁 (History API Resilience)
**目的：验证原 GetX 的“后退死锁”顽疾是否已被根除。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 用户已登录。 |
| 操作步骤 | 1. 在首页点击动态进入【详情页】。<br>2. 在详情页点击头像进入【个人主页】。<br>3. 在个人主页点击粉丝列表进入【关注页】。<br>4. 高频次点击浏览器的“后退”键 3 次。<br>5. 再点击浏览器的“前进”键 2 次。 |
| 预期结果 | 页面的进退必须和浏览器的 URL 瞬间同步，不能出现任何白屏、堆栈错乱或点击无反应的死锁现象。 |
| 实际结果 | ⬜ 待测试 |

### TC-V3-01-03 嵌套路由 Tab 状态持久化 (Stateful Retention)
**目的：验证基于 StatefulShellRoute 的底栏切换是否能够保持子页面的滚动和业务状态。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 应用处于首页（Tab 0）。 |
| 操作步骤 | 1. 在【首页】向下滚动一段距离。<br>2. 点击底栏的【个人中心】（Tab 2）。<br>3. 在个人中心执行某个弹窗操作后关闭。<br>4. 点击底栏重新切回【首页】（Tab 0）。 |
| 预期结果 | 首页必须维持在刚才滚动的具体位置，且不会触发重新拉取 API 的全量刷新（平滑切换）。 |
| 实际结果 | ⬜ 待测试 |

### TC-V3-01-04 全局弹窗上下文兜底 (Context Recovery)
**目的：验证刚才紧急修复的 Get.key 兜底方案是否在所有非路由场景中生效。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 在任意 Tab 页面。 |
| 操作步骤 | 1. 在侧边栏点击【赞助支持】，验证 `Get.dialog` 是否弹出。<br>2. 点击【退出登录】，验证是否弹出确认框。<br>3. 在某个没有 Context 的 GetX Controller 中触发 `Get.snackbar`（如点赞失败/评论成功）。 |
| 预期结果 | 1. 所有 Dialog 和 BottomSheet 必须正常阻断弹出。<br>2. 所有 Snackbar 必须在顶部正常滑出，没有任何静默报错。 |
| 实际结果 | ⬜ 待测试 |

### TC-V3-01-05 路由传参正确性 (Extra Arguments)
**目的：验证重构脚本将 `arguments` 转换为 `extra` 后的强类型传参是否正确。**
| 项目 | 内容 |
|------|------|
| 前置条件 | 存在有数据的动态列表。 |
| 操作步骤 | 1. 点击一条动态卡片，触发传入 Article 对象的路由跳转（`appRouter.push('/article/id', extra: article)`）。<br>2. 观察详情页的初始渲染。 |
| 预期结果 | 详情页应当瞬间渲染出传递的 Article 数据（而非从网络重新加载呈现骨架屏）。 |
| 实际结果 | ⬜ 待测试 |
